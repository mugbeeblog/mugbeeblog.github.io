# Gemini Code Assist Style Guide

## 基本方針

- 日本語で記述せよ
- MVPを最優先し、「完成させること」を第一目的とする
- レビューは実装を前に進めるためのものであり、完成を遅らせる指摘は最小限に抑える

---

## レビュー時の最重要原則

### 1. MVP厳守ルール

以下は 原則レビュー対象外 とし、issueとしてのみ記載せよ
（PR差し戻し・必須修正を要求してはならない）

- a11y（アクセシビリティ）
- レスポンシブ対応
- 厳密な型定義
- 網羅的なエラーハンドリング
- 過剰なコンポーネント分割
- リファクタリング全般
- 将来拡張を見越した設計

---

### 2. 削減提案の原則

- 「緊急性: 必須」以外は削減提案のみ行う
- 削減提案とは以下を指す：
  - 実装量を減らす
  - 複雑さを下げる
  - 処理や責務を統合する

- 削減できない場合のみ issue として記載せよ

---

### 3. 指摘の分類ルール（必須）

すべての指摘に対応区分を明示せよ。

#### 対応区分（必須）

- [緊急性: 必須]
  このPRで対応しないと 完成しない / 破綻する / 動かない
  例：構文エラー、ビルドエラー、明らかな論理エラー、重大なセキュリティ脆弱性

- [対応: issue]
  MVP完成には不要。後続対応でよいもの
  例：パフォーマンス最適化、可読性向上、将来の拡張性考慮、UI/UXの微調整

---

## 設計・実装原則（やりすぎ防止）

以下は「守る」ものではなく 抑制のための指針 として扱え。

- DRY（Don't Repeat Yourself）
- SRP（Single Responsibility Principle）
- KISS（Keep It Simple, Stupid）
- YAGNI（You Aren't Gonna Need It）
- MVP（Minimum Viable Product）
- 整合性チェック（Consistency Check）
- 慣例（de facto standard）
